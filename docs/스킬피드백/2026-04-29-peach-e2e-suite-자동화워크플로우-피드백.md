## 2. 해결 방법 / 우회 전략

### 문제 #1: suite 완료 보고 형식

**원인**: 보고 형식이 없으니 AI가 실행 로그 전체를 출력하거나 임의 형식으로 보고.
사용자는 핵심(몇 개 통과, 신규 PK, AI 판단)만 보고 피드백을 주면 됨.

**해결**: 표준 보고 형식 정의.

```
✅ suite 완료 — {기능명} ({통과수}/{전체수})

| 시나리오 | 결과 | 비고 |
|---|---|---|
| 사용자 01~04 | ✅ 전체 통과 | |
| 관리자 01 | ✅ | {PK_FIELD}=1777 |
| 관리자 02~04 | ✅ 전체 통과 | |

AI 판단:
- 자동 수정: {있으면 "XX 수정 (N회)" / 없으면 "없음"}
- 잔여 이슈: {있으면 1줄 / 없으면 "없음"}

팀장 확인 필요: {있으면만 표시, 없으면 이 줄 생략}
```

### 문제 #2: AI 자율 처리 범위

**원인**: 현재는 "3회 실패 → 중단"만 명시. 성공 케이스에서도 무엇을 자율 처리하고
무엇을 사용자에게 확인받을지 기준이 없음.

**해결**: 명확한 자율/확인 분류표.

| 상황 | AI 자율 처리 | 사용자 확인 필요 |
|---|---|---|
| 시나리오 실행 | ✅ | |
| 오류 원인 분석 | ✅ | |
| 시나리오 코드 수정 (3회 이내) | ✅ | |
| 3회 수정 후도 실패 | | ✅ |
| suite 전체 통과 | ✅ 보고 후 종료 | ✅ 피드백 |
| 일부 실패 + 자동수정 성공 | ✅ 보고에 수정내역 포함 | ✅ 피드백 |
| 커밋·푸시 | | ✅ 항상 |
| 시나리오 파일 신규 생성/삭제 | | ✅ 항상 |

### 문제 #3: 프로그램 변경 후 재검증 모드

**원인**: 소스 변경 후 "전체 suite 재실행"은 비효율. 변경된 파일과 관련된 시나리오만
선별 실행해야 토큰·시간 절약.

**해결**: `recheck` 모드 추가.

```bash
# 변경된 파일 감지
git diff --name-only HEAD~1 HEAD -- src/

# 관련 시나리오 선별 규칙
# - DAO/Service 변경 → 해당 모듈 전체 시나리오
# - Skin(insert/update) 변경 → 등록/수정 시나리오
# - Controller 변경 → 해당 URL 관련 시나리오
```

---

### 문제 #4/#5: state.json 초기화

**원인**: 이전 suite 실행에서 생성된 state.json의 PK가 남아있으면 다음 실행에서
이미 수정된(혹은 삭제된) 데이터로 테스트해 오탐 발생.

**해결**: suite run 시작 전 state.json 초기화 옵션.

```bash
# suite run 전 자동 초기화
rm -f e2e/.tmp/e2e_*_state.json
echo "🗑️ state.json 초기화 완료"
```
