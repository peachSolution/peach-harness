## 2. 해결 방법 / 우회 전략

### 문제 #1: 통합 판단 근거 부재

**원인**: `peach-team-e2e`는 이미 `setup + scenario + suite` 패턴을 흡수한 통합 레이어라고 설명하지만, `browse/scenario/suite`를 완전히 병합하지 않는 이유는 별도 의사결정 기록으로 정리되어 있지 않다.

**해결**: 스킬 통합이 아니라 스킬 간 호출 후크를 보강한다는 방향을 문서화한다.

확인한 근거:

```bash
nl -ba skills/peach-team-e2e/SKILL.md | sed -n '1,120p;430,455p'
```

주요 근거:

- `peach-team-e2e` 8행: `peach-e2e-setup + peach-e2e-scenario + peach-e2e-suite` 패턴 흡수 명시
- `peach-team-e2e` 28행: 기존 Tier 2 스킬은 단계별 단독 호출용으로 유지한다고 명시
- `peach-team-e2e` 440~450행: 관련 스킬과 SoT 관계 명시

### 문제 #2: browse 종료 후 시나리오화 연결 부족

**원인**: `peach-e2e-browse`는 단독 탐색/검증 역할이 명확하지만, 탐색 결과가 반복 검증 가치가 있을 때 `peach-e2e-scenario`로 넘기는 종료 기준이 없다.

**해결**: `peach-e2e-browse`의 결과 보고 단계에 아래 판단을 추가한다.

```markdown
탐색 결과가 반복 검증 가치가 있으면, 마지막 보고에 다음 항목을 포함한다:
- 시나리오화 가능 여부
- 재사용 가능한 셀렉터/URL/검증 포인트
- 추천 다음 단계: `/peach-e2e-scenario create ...`
```

### 문제 #3: scenario 실패 시 browse 폴백 명시 부족

**원인**: `peach-e2e-scenario`는 DOM 선조사와 `agent-browser eval` 역할을 갖고 있지만, 자동수정 루프 실패 시 `peach-e2e-browse`의 의사결정 트리를 재사용한다는 연결이 약하다.

**해결**: 자동수정 루프 판단 기준에 아래 원칙을 추가한다.

```markdown
셀렉터 불일치, SPA 입력 이벤트, iframe, Flutter/Canvas 의심이면
`peach-e2e-browse` 의사결정 트리로 내려가 DOM/상호작용 패턴을 재확인한 뒤 시나리오를 수정한다.
```

### 문제 #4: suite 실패 시 실패 Step 단위 하강 원칙 강화

**원인**: `peach-e2e-suite`는 Step 실패 시 중단한다고 되어 있으나, 다음 작업 경로가 "실패 Step을 scenario 단위로 재실행/수정"이라는 점이 더 명확해질 수 있다.

**해결**: 단계별 실패 처리 표에 아래 문구를 보강한다.

```markdown
suite 실패는 전체 suite를 반복하지 않고, 실패한 Step의 단위 시나리오를 먼저 `peach-e2e-scenario run` 기준으로 재현/수정한다.
수정 완료 후 suite를 재개한다.
```
