## 1. 상위 방향

피치 하네스의 상위 방향은 **AI를 오래 자율 개발시키는 것**이다.

여기서 자율 개발은 AI가 기획부터 임의로 판단한다는 뜻이 아니다. 사람은 Spec, 기획, 디자인, 정책 판단에 의도를 충분히 반영하고, AI는 그 기준을 바탕으로 개발과 검증 루프를 최대한 자율 수행한다.

즉, 피치 하네스는 다음 역할 분리를 전제로 한다.

| 구간 | 주도 | 이유 |
|------|------|------|
| 기획, 디자인, 개발 구체화 | 사람 + AI 대화 | 사람의 암묵지, 정책 판단, 제품 의도 반영이 핵심 |
| 개발, TDD, 연결 검증 | AI 자율 수행 | 기준이 명확하면 반복 구현과 코드 검증은 자동화 효과가 큼 |
| E2E, 미스매치 분류, 보완 루프 | AI 자율 수행 + 필요한 지점만 사람 확인 | 사용자 흐름 검증과 반복 보완은 하네스가 강한 영역 |

핵심 문장:

> 사람은 빡센 기획과 구체화로 기준을 만들고, AI는 그 기준을 바탕으로 개발과 검증을 오래 자율 수행한다.

### 용어 구분

| 용어 | 의미 | 위치 |
|------|------|------|
| 오래 자율 개발 | AI가 긴 시간 동안 개발과 검증을 사람 개입 없이 이어갈 수 있게 하는 상위 목표 | 하네스 전체 방향 |
| 대규모 작업의 1차 완성도 | 첫 실행에서 TDD/타입/빌드/주요 연결/핵심 흐름을 최대한 통과시키는 실행 전략 | team-dev/e2e 운영 전략 |
| 빡센 기획 | 사람의 의도, 정책, 상태, 권한, 예외, 화면 흐름을 Spec/schema/ui-proto로 구체화하는 앞단 작업 | gen-spec/ui-proto 이전과 중간 검토 |

이 셋은 같은 말이 아니다. 오래 자율 개발은 목표이고, 1차 완성도 극대화는 그 목표를 현실적으로 달성하기 위한 전략이며, 빡센 기획은 그 전략의 입력 품질을 만든다.

## 2. 핵심 전략: 대규모 작업의 1차 완성도 극대화

오래 자율 개발은 상위 목표이고, 대규모 작업의 1차 완성도 극대화는 그 목표를 현실화하는 운영 전략이다.

대규모 작업에서 목표는 한 번에 무조건 100% 완료가 아니다. 첫 실행에서 TDD, 타입, 빌드, 주요 연결, 핵심 흐름이 대부분 통과하는 상태까지 끌어올리고, 남은 결함은 자동 검증과 보완 루프로 줄인다.

이를 위해 개발 전에 다음 기준을 충분히 준비한다.

- Spec: TEST_ID, 권한, 상태 전이, 오류 케이스, API 기대값
- DB/schema: 테이블, 컬럼, 상태값, 인덱스, 마이그레이션 기준
- UI Proto: 화면 목록, 주요 액션, 완료/오류 상태, TEST_ID 매핑
- 공통 자원: 공통 상수, 공통 컴포넌트, 모듈 경계
- 검증 기준: TDD, Contract Gate, E2E Evidence, 잔여 리스크 분리

## 3. 표준 플로우와 역할

표준 플로우는 다음 순서를 기본으로 한다.

```text
peach-gen-spec
  → peach-gen-db
  → peach-team-ui-proto
  → peach-team-dev
  → peach-team-e2e
```

| 단계 | 핵심 역할 | 완료 기준 |
|------|----------|----------|
| `peach-gen-spec` | 요구사항을 구현·검증 가능한 기준으로 정제 | TEST_ID, 권한, 상태 전이, 오류 케이스, API 기대값이 드러남 |
| `peach-gen-db` | 확정 schema/migration 기준 생성 | team-dev가 참조할 DB 구조가 준비됨 |
| `peach-team-ui-proto` | 화면 해석과 사용자 흐름 기준 구체화 | 화면 목록, 주요 액션, 완료/오류 상태, TEST_ID 매핑이 남음 |
| `peach-team-dev` | 구현, TDD, lint/build, Contract Gate | 코드 수준 검증과 주요 연결이 통과하고 잔여 리스크가 정리됨 |
| `peach-team-e2e` | 실제 사용자 흐름과 기획 부합 검증 | E2E Evidence, 미스매치 분류, 자동 보완/위임 이력이 남음 |

ui-proto는 필수 단계가 아니라 품질 증폭 장치다. 단순 CRUD, 백엔드 중심 작업, 기존 화면의 작은 수정은 Spec-only로 진행할 수 있다. 신규 화면이 많거나 다단계 흐름, 권한/상태 분기, E2E 고품질 검증이 필요하면 ui-proto를 권장한다.

## 4. 스킬별 방향

### 4-1. peach-team-ui-proto

`peach-team-ui-proto`는 오래 자율 개발의 실행 주체가 아니다. 이 스킬의 핵심은 **팀 자동화보다 기획 구체화 산출물 품질**이다.

따라서 ui-proto는 다음 산출물을 명확히 남겨야 한다.

- `_spec.md` 원본
- 화면 목록
- 주요 액션
- 완료 상태와 오류 상태
- Spec `TEST_ID`와 화면/액션 매핑
- Spec 미반영 또는 검증 불가 항목

Spec 입력 시 `proto-ui-dev`와 `proto-ui-qa`를 둘 수 있지만, 그 목적은 팀 자동화 자체가 아니다. 목적은 `peach-team-dev`와 `peach-team-e2e`가 기준을 잃지 않도록 화면 해석과 사용자 흐름 기준을 선명하게 만드는 것이다.

기획자가 직접 구체화하는 경우에는 단독/대화형 흐름이 자연스럽다. 이 구간은 사람의 의도가 강하게 들어가므로 AI 팀이 임의로 내용을 증폭하지 않도록 주의한다.

### 4-2. peach-team-dev와 peach-team-e2e

`peach-team-dev`와 `peach-team-e2e`는 사용 경험상 하나의 납품 흐름으로 묶는다. 다만 내부 역할과 검증 독립성은 분리한다.

```text
개발-검증 납품 흐름
  1. peach-team-dev
     - 구현
     - TDD
     - lint/build
     - API-Store Contract Gate
     - TEST_ID 구현 매핑
     - E2E 잔여 리스크 정리

  2. peach-team-e2e
     - 사용자 흐름 검증
     - ui-proto/Spec 부합 검증
     - 미스매치 분류
     - 시나리오 오류 자동 보완
     - 명확한 코드 문제는 peach-team-dev로 위임
```

하나의 흐름처럼 오케스트레이션하되, `team-e2e`가 구현 컨텍스트에 종속되어 검증 독립성을 잃으면 안 된다. E2E에서 발견한 로직 버그는 직접 고치지 않고 `peach-team-dev`로 위임한다.

## 5. 핵심 게이트

| 게이트 | 위치 | 목적 |
|--------|------|------|
| Pre-flight Gate | 개발 전 | Spec/UI Proto/DB/권한/공통 자원 충돌 사전 제거 |
| Spec Coverage Gate | Spec 이후 | 요구사항이 TDD/E2E 검증 항목으로 연결됐는지 확인 |
| Proto Flow Gate | UI Proto 이후 | 화면과 액션 흐름이 Spec 기능과 연결됐는지 확인 |
| API-Store Contract Gate | team-dev | 백엔드 응답과 Store/UI 연결 정합성 확인 |
| Build Gate | team-dev | 타입, lint, build 통과 확인 |
| E2E Evidence Gate | team-e2e | 실제 사용자 흐름 실행 증거 확보 |
| Residual Risk Gate | 완료 전 | 남은 리스크와 검증 불가 항목 분리 |

게이트의 목적은 AI를 멈추게 하는 것이 아니라, 기준이 명확한 문제는 자동 보완하고 기준이 모호한 문제만 사람에게 올리기 위한 분류 장치다.

## 6. 사람 확인 기준

AI가 자동 처리해도 되는 항목:

- 타입 불일치
- API 응답 키 불일치
- Store 매핑 누락
- UI 바인딩 누락
- E2E 셀렉터 오류
- 대기 조건 오류
- 명확한 Spec 위반
- 명확한 UI Proto 불일치

사람 확인이 필요한 항목:

- Spec과 UI Proto 충돌
- 권한, 데이터 소유 범위, 상태 전이 정책 판단
- DB schema 변경
- 공통 모듈 또는 공통 컴포넌트 변경
- PRD에는 있으나 Spec에 없는 요구
- 검증 기준 자체가 모호한 경우
- 반복 실패 상한 도달

---

## 7. 금지 패턴

아래 패턴은 하네스 방향과 맞지 않는다.

- Spec 또는 ui-proto가 모호한데 team-dev/e2e가 임의 해석으로 계속 수정한다.
- ui-proto 단계에서 AI 팀이 사람의 의도보다 요구사항을 과도하게 증폭한다.
- team-dev가 E2E 사용자 흐름 판정을 자체 완료 기준으로 끌고 온다.
- team-e2e가 본 프로젝트 코드를 직접 수정한다.
- PRD에는 있으나 Spec에 없는 요구를 바로 구현한다.
- DB/schema, 권한 정책, 공통 모듈 변경을 사용자 확인 없이 자동 처리한다.
- 배포되는 스킬 본문에서 하네스 저장소의 `docs/` 문서를 직접 참조한다.

---

## 8. 스킬 수정 판단 기준

스킬을 수정할 때는 다음 기준을 우선한다.

1. 앞단 스킬은 사람의 의도를 더 잘 구체화하는 방향으로 강화한다.
2. 개발/검증 스킬은 AI가 오래 자율 수행할 수 있도록 입력 기준, 상태 기록, 검증 게이트를 강화한다.
3. `peach-team-ui-proto`는 팀 자동화보다 산출물 품질과 후속 스킬 입력 품질을 우선한다.
4. `peach-team-dev`와 `peach-team-e2e`는 하나의 납품 흐름으로 연결하되, 구현과 검증의 컨텍스트는 분리한다.
5. 기준이 모호한 항목은 자동 보완 루프로 끌고 가지 않고 `blocked` 또는 `*_REQUIRED`로 분리한다.
6. 스킬에 필요한 상위 정책은 SKILL.md 본문 또는 스킬 내부 `references/`에 포함한다.
7. 하네스 저장소 문서의 번호와 구조가 바뀌어도 배포된 스킬이 끊기지 않도록 설계한다.
