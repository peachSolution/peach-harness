# peach-review-ux 보강 분석

> 작성일: 2026-05-02
> 대상: `skills/peach-review-ux/`
> 분석 방식: `peach-team-analyze` 방식의 3관점 병렬 검토 + PM 종합
> 참고: Laws of UX(https://lawsofux.com), GeekNews 요약(https://news.hada.io/topic?id=29034)

---

## 1. 결론

`peach-review-ux`는 별도 스킬로 유지하는 것이 맞다. 다만 독립 호출만 기대하면 실제 워크플로우에서 누락될 가능성이 높고, UX 법칙이 추상 지적으로 흐를 위험이 있다.

따라서 1차 보강 방향은 다음이다.

1. 기존 `proto-ui-qa`, `frontend-qa`, `team-e2e`에는 아직 통합하지 않는다.
2. `peach-review-ux` 자체에 근거 레벨, 판정 매핑, 후속 전달 포맷을 추가한다.
3. UX 법칙은 판정 기준이 아니라 근거 라벨로만 사용한다.
4. 피치 백오피스에서는 반복 업무 효율, 데이터 정확성, 스캔성, 오류 방지를 최상위 기준으로 둔다.

| 역할 | 담당 |

| UX 법칙 분석 | Laws of UX/GeekNews 요약 중 피치 백오피스에 유효한 법칙 선별 |

## 3. 핵심 발견

### 3-1. 독립 스킬은 필요하지만 누락 위험이 크다

현재 표준 워크플로우는 `peach-gen-spec → peach-gen-ui-proto → peach-team-dev → peach-team-e2e` 4단계다. `peach-review-ux`는 이 흐름에 포함되지 않으므로 사용자가 명시적으로 호출하지 않으면 빠질 가능성이 높다.

하지만 바로 통합하면 모든 UI 작업의 QA가 주관적 휴리스틱에 끌려갈 수 있다. 초기에는 독립 스킬로 실사용 데이터를 모으고, 반복적으로 효과가 확인된 체크만 통합 후보로 승격하는 편이 안전하다.

### 3-2. Laws of UX 전체를 강제하면 false positive가 커진다

Laws of UX는 리뷰 관점으로는 유용하지만, “법칙 위반 = 수정 필요”로 쓰면 안 된다. 특히 백오피스는 숙련 사용자의 반복 속도와 정보 밀도가 중요하므로, 일반 소비자 서비스 UX 조언을 그대로 적용하면 오히려 작업 효율이 떨어질 수 있다.

예:
- 필터가 많다는 이유만으로 접으면 숙련자의 반복 검색 속도가 떨어질 수 있다.
- 위험 액션 확인을 과도하게 넣으면 반복 업무 흐름이 끊길 수 있다.
- 모든 주요 버튼을 크게 만들면 테이블 스캔성이 떨어질 수 있다.

### 3-3. 현재 스킬의 부족점

| 부족점 | 영향 | 보강 방향 |
|--------|------|----------|
| QA 판정 체계 없음 | 기존 QA 스킬과 연결하기 어려움 | `REJECTED 후보 / CONDITIONAL 후보 / 권고` 매핑 추가 |
| 근거 레벨 없음 | 화면 근거 없는 추상 지적 가능 | 코드/DOM/스크린샷/Spec/추정 레벨 구분 |
| 후속 전달 포맷 없음 | 리뷰 후 수정 작업으로 이어지기 어려움 | 대상 스킬, owner, E2E 포인트 명시 |
| 최소 통합 체크 없음 | 나중에 어떤 항목을 통합할지 불명확 | 5개 smoke check 정의 |
| 오용 방지 문구 부족 | 법칙의 기계적 적용 위험 | 법칙별 오용 방지 규칙 추가 |

## 4. 피치 백오피스 1순위 UX 법칙

아래 법칙은 `peach-review-ux`의 핵심 검토 기준으로 유지한다.

| 우선 | 법칙 | 피치 적용 |
|------|------|----------|
| 1 | 인지 부하 (Cognitive Load) | 화면 이해 비용, 정보 과밀, 판단 수 검토 |
| 2 | 청킹 (Chunking) | 폼/검색/상세/첨부/권한 정보 그룹화 |
| 3 | 근접의 법칙 (Proximity) | 라벨-입력, 필드-오류, 버튼-대상 관계 |
| 4 | Jakob의 법칙 + 멘탈 모델 | 백오피스 관습과 흐름 예측 가능성 |
| 5 | Hick의 법칙 + 선택 과부하 | 필터, 버튼, 탭, 일괄 액션 과다 노출 |
| 6 | Fitts의 법칙 | 클릭 대상 크기와 간격 |
| 7 | Doherty 임계값 | 저장/검색/업로드/일괄 처리 피드백 |
| 8 | 선택적 주의 + Von Restorff | 핵심 액션, 위험 액션, 오류 구분 |
| 9 | Peak-End 규칙 | 저장/실패/완료 후 결과 명확성 |
| 10 | 작업 기억 + Zeigarnik | 이전 조건, 선택 대상, 미저장 변경, 실패 항목 표시 |

## 5. 보조 법칙

아래 법칙은 핵심 판정 기준이 아니라 보조 근거로만 사용한다.

| 법칙 | 제한적 사용처 |
|------|--------------|
| 공통 영역 | 섹션 경계, 검색/목록/상세 영역 구분 |
| 유사성 | 상태 뱃지, 버튼, 같은 기능의 시각 일관성 |
| 균일 연결 | 선택 항목과 상세 패널, 단계 흐름 연결 |
| 순서 위치 효과 | 컬럼/버튼/메뉴 순서 검토 |
| Postel의 법칙 | 입력 포맷 보정, 오류 메시지, 저장값 표준화 |
| 능동적 사용자의 역설 | 빈 상태, 첫 행동, 오류 복구 안내 |
| 목표 경사 효과 | 업로드, 일괄 처리, 다단계 폼 |
| Tesler의 법칙 | 복잡한 업무 규칙을 UI가 어디까지 흡수할지 판단 |
| Occam의 면도날 | 불필요한 장식, 단계, 설정 제거 |
| Pareto 원칙 | 반복 업무 중심의 개선 우선순위 |
| 심미적 사용성 효과 | 시각 정돈의 보조 근거. 작업 성공률 검증을 대체하지 않음 |

## 6. 오용 방지 가드레일

| 법칙 | 오용 | 방지 문구 |
|------|------|----------|
| Miller의 법칙 | 항목을 무조건 7개 이하로 제한 | 숫자 제한이 아니라 의미 단위 그룹화 여부를 본다 |
| 심미적 사용성 효과 | 예쁘면 사용성도 좋다고 판단 | 시각 완성도와 작업 성공률은 별도로 검증한다 |
| Von Restorff | 모든 중요 요소를 강조 | 강한 강조는 핵심 액션/위험/오류 등 소수에 제한한다 |
| Hick/선택 과부하 | 숙련자용 기능을 숨김 | 자주 쓰는 기본값은 노출하고 드문 옵션만 접거나 묶는다 |
| Fitts | 모든 버튼을 크게 만듦 | 조작 빈도와 위험도에 따라 크기와 간격을 조정한다 |
| Peak-End | 완료 토스트/모달 남발 | 완료 결과는 명확하되 반복 작업을 방해하지 않는다 |
| Goal-Gradient/Zeigarnik | 단순 CRUD에도 진행률/미완료 표시 추가 | 다단계·대량 처리·미저장 변경처럼 진행/손실 리스크가 있을 때만 쓴다 |
| Pareto | 저빈도 고위험 업무를 무시 | 우선순위 판단에만 쓰고 실패 비용이 큰 저빈도 작업은 별도 검토한다 |

## 7. 추후 통합 후보

### 7-1. proto-ui-qa

가장 먼저 통합할 수 있는 위치다. 이미 “Spec 부합 + 시각/UX 검증” 책임이 있으므로, 전체 법칙이 아니라 `UX smoke check` 5개만 선택 항목으로 넣는 것이 적절하다.

후보 5개:
1. 핵심 액션이 보조 액션과 구분되는가
2. 위험 액션이 일반 액션과 구분되는가
3. 저장/검색/업로드 피드백이 있는가
4. 오류가 필드 근처에 있고 다음 행동을 알려주는가
5. 완료 후 결과와 다음 행동이 명확한가

### 7-2. frontend-qa

기계 검증 게이트에 바로 넣으면 위험하다. UX 리스크는 빌드 실패와 같은 강도의 `REJECTED`가 아니라, 기계 검증 통과 후 `CONDITIONAL` 근거로 보고하는 편이 안전하다.

### 7-3. team-e2e

E2E에는 UX 법칙 전체가 아니라 사용자 흐름 결과 검증만 붙인다.

적합 항목:
- 클릭 후 로딩/비활성화/진행 상태가 보이는가
- 저장/실패 후 결과가 확인 가능한가
- 완료 후 사용자가 다음 행동을 알 수 있는가
