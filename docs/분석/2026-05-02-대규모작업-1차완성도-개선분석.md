# 대규모 작업 1차 완성도 개선 분석

> 작성일: 2026-05-02  
> 목적: 많은 Spec을 기반으로 한 대규모 작업을 처음 실행할 때 최대한 높은 완성도까지 끌어올리기 위한 개선 방향 정리  
> 기준 문서: 대규모 일괄 개발 리스크 분석, 내부 프로젝트 운영 모델 분석

---

## 1. 핵심 결론

목표를 `자율 개발`로 잡으면 범위가 너무 넓어진다. 지금 필요한 방향은 더 구체적으로 **대규모 작업의 1차 완성도 극대화**다.

즉, AI가 모든 것을 완전히 알아서 판단하게 만드는 것이 아니라, 개발 전 기준을 충분히 정리하고 `team-dev`, `team-e2e`가 처음 실행에서 누락·불일치·재작업을 최대한 줄이도록 만드는 것이다.

핵심은 다음이다.

1. 개발 전에 Spec, UI Proto, DB, 권한, 공통 자원 정합성을 먼저 본다.
2. 큰 작업을 기능 큐로 나누고, 기능별 입력과 완료 기준을 명확히 한다.
3. `team-dev`에서 BE·Store·UI 연결 품질을 최대한 끌어올린다.
4. `team-e2e`에서 남은 결함을 원인별로 분류하고 보완한다.
5. 사람 확인은 기준 충돌, 정책 판단, 공통 구조 변경처럼 꼭 필요한 지점으로 제한한다.

---

## 2. 1차 완성도의 의미

여기서 1차 완성도는 “한 번 실행해서 무조건 100% 완료”가 아니다. 대규모 작업에서는 한 번에 완전한 완료를 보장하기 어렵다.

현실적인 목표는 다음이다.

| 구분 | 의미 |
|------|------|
| 높은 1차 완성도 | 개발 직후 TDD, 타입, 빌드, 주요 연결, 핵심 흐름이 대부분 통과하는 상태 |
| 낮은 1차 완성도 | 파일은 생성됐지만 연결 누락, 타입 불일치, Spec 미반영, E2E 실패가 많이 남은 상태 |
| 목표 | 첫 실행에서 운영 가능한 구조에 가깝게 만들고, 남은 결함은 자동 검증·보완 루프로 줄인다 |

따라서 품질 목표는 다음처럼 잡는다.

1. Backend TDD가 통과한다.
2. Store와 API 타입이 맞는다.
3. UI가 Store와 실제 응답 구조에 맞게 연결된다.
4. 주요 사용자 흐름이 E2E로 실행 가능하다.
5. 권한, 상태 전이, 오류 케이스가 누락되지 않는다.
6. 검증 불가 항목은 숨기지 않고 별도 보고된다.

## 3. 현재 방식의 한계

### 3-1. 큰 작업을 한 세션에 넣으면 품질이 떨어진다

Spec, UI Proto, TDD, Store, UI, E2E, Ralph Loop 피드백이 한 세션에 계속 누적되면 후반 작업에서 지침 누락과 판단 품질 저하가 발생한다.

필요한 변화:

- 한 번에 모두 처리하는 구조가 아니라 기능 큐 기반으로 처리한다.
- 각 기능의 상태와 실패 원인을 기록한다.
- 실패한 기능만 다시 처리한다.

### 3-2. 개발 전 정합성 확인이 부족하다

개발을 시작한 뒤에 Spec 충돌, UI Proto 누락, DB 필드 부족, 공통 상수 충돌이 발견되면 재작업이 커진다.

필요한 변화:

- 개발 전에 Pre-flight를 수행한다.
- Spec, UI Proto, DB, 권한, 공통 자원을 먼저 대조한다.
- 검증 가능한 항목과 검증 불가 항목을 구분한다.

### 3-3. BE·Store·UI 연결 검증이 약하다

TDD와 build가 통과해도 백엔드 응답과 Store 타입, UI 바인딩이 미묘하게 어긋날 수 있다. 이 문제는 보통 E2E에서 늦게 발견된다.

필요한 변화:

- `team-dev`에 API-Store Contract Gate를 추가한다.
- API 응답, 타입, Store 변환, UI 사용 필드를 대조한다.

### 3-4. E2E 실패가 수정 루프로 바로 연결되지 않는다

E2E 실패가 코드 문제인지, UI Proto 불일치인지, 시나리오 오류인지 분류되지 않으면 사람 확인이 늘어난다.

필요한 변화:

- `team-e2e`가 실패를 원인별로 분류한다.
- 시나리오 오류는 자동 수정하고, 명확한 코드 문제는 안전 조건을 확인한 뒤 `team-dev`로 자동 위임한다.
- 기준이 모호한 경우만 사람에게 확인한다.

## 4. 개발 전에 집중해서 준비할 것

대규모 작업의 품질은 개발 전에 거의 결정된다. AI가 좋은 1차 결과를 내기 위해서는 다음 입력이 정리되어 있어야 한다.

### 4-1. Spec 준비

Spec은 단순 요구사항 목록이 아니라 검증 기준이어야 한다.

필수 항목:

| 항목 | 설명 |
|------|------|
| 기능 목록 | 개발할 기능 단위 |
| 권한 경계 | 누가 어떤 데이터에 접근할 수 있는지 |
| 상태 전이 | 상태값과 변경 가능 조건 |
| 오류 케이스 | 입력 검증, 권한 실패, 외부 실패, 서버 오류 |
| API 기대값 | 요청, 응답, 페이지네이션, 에러 응답 |
| 데이터 변화 | 저장, 수정, 삭제 후 기대 DB 상태 |
| TEST_ID | 각 요구사항을 추적할 수 있는 ID |

### 4-2. UI Proto 준비

UI Proto는 화면의 기준이다. 단순 목업이 아니라 사용자 흐름을 읽을 수 있어야 한다.

필수 항목:

| 항목 | 설명 |
|------|------|
| 화면 목록 | 목록, 상세, 등록, 수정, 모달, 빈 상태 |
| 주요 액션 | 검색, 저장, 승인, 반려, 업로드, 다운로드 등 |
| 화면 전환 | 페이지 이동, 모달, 탭, 단계 흐름 |
| 완료 상태 | 저장 후 메시지, 목록 반영, 다음 행동 |
| 오류 상태 | 필드 오류, 권한 차단, 실패 안내 |
| Spec 매핑 | 각 화면이 어떤 Spec 기능을 구현하는지 |

### 4-3. DB와 공통 자원 준비

대규모 작업에서는 DB와 공통 자원 충돌이 재작업의 큰 원인이다.

확인 항목:

- 필요한 테이블과 컬럼이 있는가
- 상태 필드와 인덱스가 충분한가
- 기존 모듈과 네이밍이 충돌하지 않는가
- 공통 상수 추가가 필요한가
- 공통 컴포넌트 수정이 필요한가
- 기존 권한/세션/조직 기준과 맞는가

---

## 5. 권장 실행 흐름

```text
Stage 0. 입력 정리
  기획 자료, 회의 내용, 기존 문서를 구조화한다.

Stage 1. Pre-flight
  Spec, UI Proto, DB, 권한, 공통 자원 정합성을 확인한다.

Stage 2. 작업 큐 생성
  큰 작업을 기능 큐로 나누고 처리 순서와 병렬 가능 범위를 정한다.

Stage 3. team-dev 1차 개발
  백엔드 TDD, Store 연결, UI 구현, lint/build, Contract Gate를 수행한다.

Stage 4. team-e2e 검증
  단위 시나리오와 통합 시나리오를 만들고 실행한다.

Stage 5. 보완 루프
  실패를 분류하고 시나리오 오류는 자동 수정, 명확한 코드 문제는 안전 조건 확인 후 자동 위임한다.
```

## 6. 스킬별 보완 수준

### 6-1. `peach-gen-spec` — 보강 수준: 매우 높음

대규모 작업의 1차 완성도는 Spec 품질에 크게 좌우된다. `peach-gen-spec`은 요구사항 문서를 만드는 수준을 넘어, 개발과 검증의 기준을 만드는 역할로 강화해야 한다.

보완 내용:

- 권한 경계 매트릭스 추가
- 상태 전이 표 추가
- 오류 케이스 표 추가
- 외부 의존성 표 추가
- API 기대값 명시
- TEST_ID 부여
- TEST_ID별 TDD/E2E 검증 위치 매핑

완성 기준:

- 각 요구사항이 구현 대상인지, TDD 대상인지, E2E 대상인지 구분된다.
- 권한·상태·오류 케이스가 누락되지 않는다.
- team-dev와 team-e2e가 같은 기준을 읽는다.

### 6-2. `peach-gen-ui-proto` — 보강 수준: 높음

UI Proto는 화면 구성뿐 아니라 사용자 흐름의 기준이어야 한다.

보완 내용:

- 화면 목록 자동 추출
- 주요 액션 목록화
- 화면 전환 흐름 정리
- 완료 상태와 오류 상태 정의
- Spec TEST_ID와 화면 매핑
- 기능 큐 생성에 필요한 화면 단위 분해

완성 기준:

- UI 구현자가 어떤 화면과 액션을 만들어야 하는지 바로 알 수 있다.
- team-e2e가 Proto를 기준으로 시나리오를 생성할 수 있다.
- Spec에는 있는데 화면이 없는 항목이 드러난다.

### 6-3. `peach-team-dev` — 보강 수준: 매우 높음

`team-dev`는 1차 완성도를 실제로 끌어올리는 핵심 스킬이다.

보완 내용:

- 기능 큐 입력 지원
- 기능별 상태 파일 생성
- 실패 기능만 재시도하는 구조
- API-Store Contract Gate 추가
- Spec TEST_ID와 구현 결과 매핑 보고
- Backend, Store, UI 연결 검증 강화
- 완료 시 E2E로 넘길 잔여 리스크 정리

완성 기준:

- Backend TDD, lint, build가 통과한다.
- Store 타입 검증이 통과한다.
- Frontend 타입, lint, build가 통과한다.
- API 응답과 Store/UI 사용 필드가 맞는다.
- 기능별로 done/blocked/qa_failed 상태가 남는다.

### 6-4. `peach-team-e2e` — 보강 수준: 높음

`team-e2e`는 남은 결함을 줄이는 검증·보완 스킬이다.

보완 내용:

- TEST_ID 기반 E2E 커버리지 확인
- Spec 위반, UI Proto 불일치, 시나리오 오류, 기준 모호 분류
- 시나리오 오류 자동수정 강화
- 명확한 코드 문제는 안전 조건 확인 후 `team-dev` 자동 위임
- E2E Evidence 보고서 강화
- 검증 불가 항목 별도 기록

완성 기준:

- 단위 시나리오와 통합 시나리오가 생성된다.
- 주요 흐름이 실제 실행된다.
- 실패 원인이 분류된다.
- 자동 보완 후 재실행된다.
- 남은 항목은 근거와 함께 보고된다.

### 6-5. `peach-qa-gate` — 보강 수준: 중간

`peach-qa-gate`는 최종 품질 증거를 수집하는 역할로 강화하면 좋다.

보완 내용:

- 기능별 TDD/lint/build 결과 수집
- Contract Gate 결과 수집
- TEST_ID 커버리지 수집
- E2E Evidence 링크 수집
- blocked 항목과 검증 불가 항목 요약

완성 기준:

- “완료”라는 말이 아니라 실제 증거 기반 완료 보고가 가능하다.

## 7. 핵심 게이트

| 게이트 | 위치 | 목적 |
|--------|------|------|
| Pre-flight Gate | 개발 전 | Spec/UI Proto/DB/권한/공통 자원 충돌 사전 제거 |
| Spec Coverage Gate | Spec 이후 | 요구사항이 TDD/E2E 검증 항목으로 연결됐는지 확인 |
| Proto Flow Gate | UI Proto 이후 | 화면과 액션 흐름이 Spec 기능과 연결됐는지 확인 |
| API-Store Contract Gate | team-dev | 백엔드 응답과 Store/UI 연결 정합성 확인 |
| Build Gate | team-dev | 타입, lint, build 통과 확인 |
| E2E Evidence Gate | team-e2e | 실제 사용자 흐름 실행 증거 확보 |
| Residual Risk Gate | 완료 전 | 남은 리스크와 검증 불가 항목 분리 |

## 8. 사람 확인이 필요한 지점

사람 확인을 줄이되 완전히 없애면 안 된다. 아래는 사람 확인이 필요한 지점이다.

| 상황 | 이유 |
|------|------|
| Spec과 UI Proto 충돌 | 기준 선택은 비즈니스 판단 |
| 권한·데이터 소유 범위 변경 | 운영 정책 영향 |
| DB 구조 변경 | 마이그레이션과 데이터 영향 |
| 공통 모듈·공통 컴포넌트 변경 | 다른 기능 영향 |
| 반복 실패 상한 도달 | 자동 보완 신뢰도 저하 |
| 검증 기준 자체가 모호함 | AI가 임의 해석하면 재작업 위험 |

반대로 아래는 자동 처리 대상으로 둔다.

- 타입 불일치
- API 응답 키 불일치
- Store 매핑 누락
- UI 바인딩 누락
- E2E 셀렉터 오류
- 대기 조건 오류
- 명확한 Spec 위반
- 명확한 UI Proto 불일치

## 9. 문서와 스킬 개선 우선순위

| 우선순위 | 대상 | 개선 내용 |
|----------|------|-----------|
| 1 | `peach-gen-spec` | TEST_ID, 권한, 상태 전이, 오류 케이스 구조화 |
| 2 | `peach-team-dev` | 기능 큐, 상태 파일, Contract Gate 추가 |
| 3 | `peach-team-e2e` | 실패 분류, 시나리오 자동수정, 코드 수정 확인 후 위임, Evidence 보고 강화 |
| 4 | `peach-gen-ui-proto` | 화면·액션·완료/오류 상태와 Spec 매핑 강화 |
| 5 | `peach-qa-gate` | 전체 증거 수집과 잔여 리스크 요약 강화 |

## 10. 최종 판단

대규모 작업을 고품질로 처리하려면 핵심은 “AI가 알아서 전부 하게 만들기”가 아니다. 핵심은 **AI가 처음부터 높은 완성도로 만들 수 있도록 입력 기준과 검증 기준을 정밀하게 준비하는 것**이다.

따라서 개선 방향은 다음 한 문장으로 정리된다.

> 대규모 작업은 한 번에 끝내는 것이 아니라, 1차 실행에서 최대한 완성도를 끌어올리고 남은 결함을 자동 검증·보완 루프로 줄이는 방식으로 설계한다.

이 방향으로 가면 사람의 역할도 바뀐다.

- 기존: 매번 결과를 확인하고 수정 지시
- 개선 후: 개발 전 기준을 승인하고, 기준 충돌이나 정책 판단만 처리

즉, 개발 속도를 높이는 핵심은 실행 단계 자동화보다 **개발 전 준비 품질과 검증 게이트 강화**다.
