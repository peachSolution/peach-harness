# 스킬 재구성 의사결정 (2026-04-27)

> 작성일: 2026-04-27
> 결정자: 박동재
> 배경: SemulineDesktop 하네스 워크플로우 패턴 분석 + ui-proto-backoffice 프로젝트 구조 검토 결과

---

## 1. 배경

기존 피치 하네스 워크플로우는 `peach-gen-spec → peach-gen-db → peach-team → peach-qa-gate` 직선형이었으나, 다음 한계가 있었다.

- **Spec 검증 시점이 늦음** — 글로 된 Spec만으로 기획자 검토 → 풀스택 구현 후 화면을 처음 본 시점에 큰 수정 발생
- **ui-proto가 워크플로우에서 고립** — `peach-gen-ui-proto` 스킬은 존재했지만 표준 플로우에 포함되지 않음
- **E2E가 워크플로우 종결과 단절** — `peach-e2e-*` 스킬이 본 개발과 별도로 호출되어 검증 기준이 흐림
- **스킬 27개로 증가** — 진입점이 불명확해 신규 사용자 진입 장벽 발생
- **리팩토링 스킬의 의문** — AI Plan 모드 + Edit으로 충분히 처리 가능한 영역에 별도 스킬 3개

ui-proto-backoffice의 `modules-task` 시간순 누적 패턴이 살아있는 모범 사례로 확인되어, 이를 표준 플로우에 흡수한다.

## 2. 핵심 결정

### 2-1. 검증 기준의 외부화 (가장 중요)

**기존 개발은 "코드가 동작하는지"만 검증**했으나, 이번 재구성으로 **"기획 의도와 부합하는지"**를 검증 기준으로 삼는다.

검증 기준 산출물:
- **ui-proto** — 시각/인터랙션 흐름의 1차 기준
- **Spec 문서** — 비즈니스 규칙/예외/데이터 정확성의 1차 기준

### 2-2. 검증 우선순위 규칙

ui-proto는 본질적으로 Mock이라 100% 구현되지 않을 수 있다. 따라서 다음 우선순위를 적용한다.

| 검증 항목 | 1차 기준 | 2차 기준 |
|----------|---------|---------|
| 화면 레이아웃, 컴포넌트 배치 | ui-proto 화면 | (없으면 Spec) |
| 사용자 인터랙션 흐름 (클릭/입력 시퀀스) | ui-proto 화면 | Spec |
| 비즈니스 규칙 (검증, 권한, 분기) | Spec | (없으면 검증 불가 → 보고) |
| 데이터 정확성 (API 응답값, DB 상태) | Spec | (없으면 검증 불가 → 보고) |
| 에러/예외 처리 | Spec | - |

**핵심 원칙**: ui-proto가 있으면 ui-proto, 없거나 모호하면 Spec, 둘 다 모호하면 보고.

### 2-3. Spec Source of Truth = ui-proto 저장소

- ui-proto 저장소의 `modules-task/{태스크폴더}/_spec.md`가 **1차 원본**
- 본 프로젝트의 `docs/spec/{년}/{월}/...`는 **사본**
- `peach-team-dev` 시작 시 자동 복사

이 구조의 이유:
- 기획자가 본 저장소 클론 없이 ui-proto에서 작업 가능
- ui-proto 1개가 여러 본 프로젝트(backoffice, www 등)에 사용 가능
- 본 프로젝트는 dev 시작 시점에 확정되므로 그때 복사가 가장 자연스러움

## 4. peach-team-dev 명세

### 4-1. 입력 모드 3가지

`peach-team-dev`는 3가지 입력 형태를 지원한다. 검증 기준의 강도가 다르므로 사용 케이스도 다르다.

| 모드 | 입력 | 검증 기준 | 적합 케이스 |
|------|------|---------|-----------|
| **표준** | Spec + DB 스키마 + proto | ui-proto 화면 + Spec | 신규 기능 (재구성 핵심 경로) |
| **Spec만** | `docs/spec/...` 본 프로젝트에 존재 | Spec | 백엔드만, UI 없는 변경 |
| **prompt** | 자연어 입력 | prompt 텍스트 | 버그 수정, 즉흥적 변경, 핫픽스 |

prompt 모드는 검증 기준이 약하므로 박동재님이 결과를 직접 확인해야 한다. 변경 규모가 크면 Spec 작성을 자동 권고한다.

### 4-2. 인자

```bash
# 표준 모드 (DB 스키마 준비 후)
/peach-team-dev mode=backend|ui|fullstack proto=<경로> [model=opus|sonnet|haiku]

# Spec만 모드 (proto 생략)
/peach-team-dev mode=backend|ui|fullstack [model=...]

# prompt 모드 (mode/proto 생략, 자연어 입력)
/peach-team-dev "버그/개선 사항 자연어 설명" [force=Y] [model=...]
```

| 인자 | 역할 |
|------|------|
| `mode` | 표준/Spec만 모드에서 필수. `backend`(API+Store), `ui`(UI만), `fullstack`(전체) |
| `proto` | 선택. ui-proto 저장소의 태스크 폴더 절대 경로 |
| `prompt` | 위치 인자. 자연어 작업 설명 (mode/proto 없을 때) |
| `force` | 선택. 큰 변경에도 Spec 권고 무시하고 강제 진행 (`force=Y`) |
| `model` | 선택. 서브에이전트 모델 override |

### 4-3. prompt 모드 동작

prompt 입력을 받으면 다음 순서로 처리한다.

1. **입력 분석**
   - 작업 규모 추정 (단일 파일 vs 다중 파일, 신규 vs 수정)
   - 영향 범위 추정 (백엔드/UI/풀스택)
   - 기존 기능 수정이면 `peach-doc-feature` 권고 여부 판단

2. **분기**
   - **소규모** (단일 파일, 명확한 패턴): 즉시 단독 모드 실행
   - **중규모** (여러 파일): 팀 모드 실행, Spec 자동 생성 제안 (선택)
   - **대규모** (모듈 추가, 다중 모듈 연동): "이건 큰 변경입니다. `/peach-gen-spec`으로 Spec 먼저 만들고 진행하는 게 안전합니다. 계속할까요?" 안내. `force=Y`이면 그대로 진행

3. **실행**
   - prompt를 임시 컨텍스트로 주입
   - 단독/팀 모드에 따라 서브에이전트 분배
   - 랄프루프 5회 (prompt 모드는 검증 기준이 약하므로 10회 비권장)

규모 추정 기준은 v1에서는 AI 자체 판단으로 둔다. 실사용 후 `peach-skill-feedback`으로 누적된 임계치를 v1.18.x 패치에서 정교화한다.

### 4-4. proto 인자 자동 처리

`proto=<경로>`가 주어지면:

1. **proto 경로 검증** — 폴더 존재, `_spec.md` 존재
2. **Spec 자동 복사** — `_spec.md`를 `docs/spec/{년}/{월}/{개발자}-{YYMMDD}-{기능명}.md`로 복사
   - 파일명은 `_task-meta.ts`에서 추출 (planner, date, title)
3. **DB 기준 확인** — `mode=backend|fullstack`이면 `api/db/schema/...` 존재 확인
4. **충돌 감지** — 본 프로젝트에 동일 spec 파일이 이미 있으면 사용자 확인:
   - (a) 덮어쓰기
   - (b) 기존 유지
   - (c) 차이점 보여주고 결정
5. **컨텍스트 주입** — 복사된 Spec + ui-proto 화면 폴더를 구현 참고/검증 기준으로 주입
6. **dev 진행**

`proto` 인자 없으면 기존처럼 `docs/spec/`의 Spec을 직접 사용.

### 4-5. 권장 패턴 — 백엔드/UI 분할

```bash
# 1. DB 기준 생성
/peach-gen-db

# 2. 백엔드 먼저 (Spec과 DB가 있으면 가능)
/peach-team-dev mode=backend

# 3. ui-proto 작성/완성 (병렬 가능)

# 4. UI 이어서 (백엔드가 안정된 상태에서 실 API 붙여 검증)
/peach-team-dev mode=ui proto=/Users/.../260427-nettem-goods
```

분할 패턴이 일괄(`mode=fullstack`)보다 권장되는 이유:
- 백엔드 TDD 안정 후 UI 검증 → 백엔드 버그 노이즈 제거
- 세션 분할로 컨텍스트 가벼워짐
- 실제 개발 흐름(백엔드와 ui-proto 병렬 진행)에 부합

작은 기능은 `mode=fullstack` 일괄도 허용.

## 5. peach-team-e2e 명세

### 5-1. 인자

```bash
/peach-team-e2e [proto=<경로>] [model=opus|sonnet|haiku]
```

### 5-2. 동작

1. **검증 기준 로드**
   - `proto` 경로의 `_spec.md`와 화면 폴더
   - 본 프로젝트의 `docs/spec/.../`

   - (a) Spec 비즈니스 규칙 위반 → 본 프로젝트 코드 수정
   - (b) ui-proto 화면 흐름과 다름 → 본 프로젝트 코드 수정
   - (c) 시나리오 오류 → 시나리오 수정
4. **보완 루프** — 최대 5/10회 (변경 규모 기반 판단)

## 6. gen-* 스킬과 team-dev의 관계 (호출 vs 공유 패턴)

team-dev는 gen-* 스킬을 **외부 호출하지 않는다.** 대신 **동일한 가이드 코드와 핵심 구현 패턴을 공유**한다.
단, `peach-gen-db`는 마이그레이션 생성/적용/스키마 추출이라는 수명주기가 다르므로 team-dev에 흡수하지 않는다. team-dev는 준비된 `api/db/schema/...`를 입력 기준으로 사용한다.

### 진짜 SSOT는 가이드 코드

```
                대상 프로젝트의 가이드 코드
                  (api/src/modules/test-data/ 등)
                  ★ 진짜 Source of Truth ★
                       ↑           ↑
                       │           │
        peach-gen-backend     peach-team-dev backend-dev-agent
        (단독 호출용 안내)      (팀 컨텍스트용 안내)
```

| 스킬 | 호출 방식 | 컨텍스트 |
|------|----------|---------|
| /peach-gen-db | DB 기준 생성 | team-dev 선행 단계 |
| /peach-gen-backend | 단독 호출 | 사용자가 직접 호출 |
| /peach-team-dev (backend-dev-agent) | 팀 컨텍스트 | 오케스트레이터가 분배 |

### 공유 패턴 방식을 채택한 이유

1. **서브에이전트 단일 컨텍스트 실행 모델** — 서브에이전트가 또 다른 스킬을 호출하면 컨텍스트 폭발
2. **랄프루프 QA 피드백 자연 주입** — 외부 스킬은 자체 워크플로우가 있어 팀 컨텍스트와 충돌
3. **드리프트 우려 작음** — 두 스킬 모두 같은 가이드 코드를 가리키므로 패턴 차이가 코드 변경 없이는 발생하지 않음
4. **이미 검증된 패턴** — 기존 팀 개발 스킬이 이 방식으로 동작한 이력이 있음

### 동기화 보장

- 공통 패턴은 가이드 코드(test-data/)로 일원화 — 이미 그렇게 되어 있음
- 페르소나의 "Bounded Autonomy" 섹션은 양쪽 동일하게 유지 (변경 시 양쪽 점검)
- Phase 2에서 작성하는 team-dev의 references는 기존 팀 개발 스킬의 references를 검토해 필요한 부분만 이전

## 7. SKILL.md 구조 정책

신규 통합 스킬은 다음 구조를 따른다.

```
skills/peach-team-dev/
├── SKILL.md              # 진입점 (개요, 인자, 호출 절차)
└── references/
    ├── connect-workflow.md    # backend/store 연결 상세
    ├── fullstack-workflow.md  # mode=fullstack 상세
    ├── prompt-mode.md         # 자연어 prompt 모드 상세
    ├── proto-sync.md     # Spec 자동 복사 절차
    ├── qa-policy.md      # QA 판정/완료 정책
    └── *-agent.md        # 서브에이전트 정의
```

이유:
- SKILL.md 비대화 방지 (진입점만 가벼움)
- 조건부 참조로 토큰 절약 (mode별로 필요한 reference만 로드)
- 기존 팀 개발 스킬의 references/ 분리 패턴과 일관성

### skill-creator 원칙 차용 (4대 가이드)

Anthropic의 `skill-creator` 스킬에서 다음 4가지 원칙을 차용한다.

| 원칙 | 내용 | 적용 |
|------|------|------|
| **Progressive Disclosure** | 메타데이터(name+description) → SKILL.md 본문 → references/의 3단계 로딩 | SKILL.md ≤ 500줄, references는 조건부 로드 |
| **description은 트리거링의 핵심** | 약간 pushy하게, 트리거 키워드 충분히 포함 | 신규 2개 스킬 description 작성 시 적용 |
| **why를 설명** | MUST 남발 대신 이유 설명. LLM은 컨텍스트 이해 능력이 충분 | 본문에서 이유 우선, 강제 어구 최소화 |
| **examples 패턴** | 핵심 동작은 입력/출력 예시로 보여줌 | proto 인자, mode 조합 등 예시 명시 |

skill-creator의 자동 평가/eval-viewer/블라인드 비교는 적용하지 않는다 (PeachSolution 전용 내부 스킬이라 박동재님 직접 검증 + `peach-skill-feedback` 누적 피드백이 더 정확).

## 8. 표준 플로우 (확정)

```
1. /peach-gen-spec              (선택, 기획자 직접 작업이면 생략 가능)
        ↓
2. /peach-gen-db                (DB 스키마/마이그레이션 기준 생성)
        ↓
3. /peach-gen-ui-proto          (Spec 입력 시 팀 모드 자동, 랄프루프 5/10)
        ↓
   기획자 검토 (브라우저 /task 페이지)
        ↓
4. /peach-team-dev proto=<경로>  (Spec 자동 복사 + 준비된 DB 기준으로 풀스택 또는 분할)
        ↓
5. /peach-team-e2e proto=<경로>  (Spec + ui-proto 종합 검증)
        ↓
   완료
```

**5단계 직선형. 신규 스킬 2개. 기존 스킬 4개 폐기.**

## 9. 사용 패턴

### 패턴 A: 표준 (Spec → ui-proto → 본 개발)

```
/peach-gen-spec → /peach-gen-db → /peach-gen-ui-proto → /peach-team-dev → /peach-team-e2e
```

### 패턴 B: 기획자 직접 (Spec 없이 ui-proto부터)

```
/peach-gen-ui-proto (단독 모드) → (필요시 사후) /peach-gen-spec → 본 프로젝트 진행
```

### 패턴 C: 분할 (백엔드 먼저, 권장)

```
/peach-gen-spec → /peach-gen-db → /peach-team-dev mode=backend
                              ↓
/peach-gen-ui-proto (병렬 진행 가능)
                              ↓
/peach-team-dev mode=ui proto=<경로> → /peach-team-e2e
```

### 패턴 D: 작은 단일 기능

```
/peach-team-3a (현행 유지)
```

### 패턴 E: 기존 기능 수정 (As-Is 분석 필요)

```
/peach-doc-feature → AI Plan 모드 → 직접 수정 또는 /peach-team-dev
```

기존 코드 의도/설계결정/테스트 함정을 먼저 보존해야 하는 경우.

### 패턴 F: 즉흥적 작업 (Spec/proto 없이, prompt 모드)

```
/peach-team-dev "버그/개선 사항을 자연어로 입력"
  ↓
team-dev가 입력 분석 → 규모에 따라 분기:
  - 소규모: 단독 모드 즉시 실행
  - 중규모: 팀 모드 + (선택) Spec 자동 생성 제안
  - 대규모: Spec 작성 권고 (사용자 확정 또는 force=Y로 진행)
```

적합 케이스:
- 작은 버그 수정 ("X 버튼 클릭 시 Y가 안 되는 문제 고쳐줘")
- 즉흥적 기능 추가 ("이 화면에 검색 필터 하나 추가해줘")
- 성능 개선 ("이 쿼리 느려, 인덱스 추가하고 N+1 해결해줘")
- 긴급 핫픽스 ("운영에서 에러 나고 있어, 바로 고쳐야 해")

주의:
- 검증 기준이 prompt 텍스트뿐이므로 결과를 직접 확인해야 한다
- 큰 변경은 Spec 권고가 자동으로 뜬다 (`force=Y`로 무시 가능)
- 매우 작은 수정은 AI Plan 모드 + Edit이 더 빠를 수 있다
