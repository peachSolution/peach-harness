# 피치 백오피스 UX 체크리스트

피치 백오피스 화면은 반복 업무, 정확한 데이터 처리, 빠른 스캔을 우선한다. 마케팅 사이트처럼 감성적 표현이나 장식적 인터랙션을 늘리지 않는다.

---

## 1. 목록 화면

### 검색 영역
- 기본 검색 조건과 고급 검색 조건이 구분되어 있는가
- 날짜 범위, 상태, 키워드처럼 자주 쓰는 필터가 앞에 있는가
- 검색/초기화 버튼이 필터 그룹과 가까운가
- 검색 후 조건이 URL 또는 화면 상태에 유지되는가

관련 법칙:
- Hick의 법칙
- 선택 과부하
- 근접의 법칙
- 작업 기억

### 테이블 영역
- 사용자가 가장 먼저 확인해야 하는 컬럼이 앞쪽에 있는가
- 상태값은 텍스트만이 아니라 색상/뱃지/라벨로 빠르게 구분되는가
- 행 클릭, 상세 버튼, 체크박스 선택의 의미가 충돌하지 않는가
- 행 액션 버튼이 너무 작거나 촘촘하지 않은가

관련 법칙:
- 선택적 주의
- 순서 위치 효과
- Fitts의 법칙
- 유사성의 법칙

### 목록 액션
- 주요 액션은 1개가 명확히 강조되는가
- 보조 액션은 같은 강도로 나열되지 않는가
- 삭제/비활성/일괄 변경 같은 위험 액션은 일반 액션과 구분되는가

관련 법칙:
- Von Restorff 효과
- Peak-End 규칙
- 인지 편향

---

## 2. 상세/등록/수정 화면

### 폼 구조
- 필드가 의미 단위로 나뉘어 있는가
- 필수값, 형식, 도움말이 입력 위치와 가까운가
- 등록/수정/상세에서 필드 순서와 명칭이 일관되는가
- 입력 수가 많으면 섹션, 탭, 단계 분리가 필요한가

관련 법칙:
- 청킹
- 공통 영역의 법칙
- 근접의 법칙
- 작업 기억

### 검증과 오류
- 오류가 필드 근처에 표시되는가
- 오류 메시지가 다음 행동을 알려주는가
- 저장 실패 후 사용자가 입력한 값이 보존되는가
- 서버 검증 오류와 프론트 검증 오류가 다른 방식으로 혼란스럽게 표시되지 않는가

관련 법칙:
- Postel의 법칙
- Peak-End 규칙
- 능동적 사용자의 역설

### 저장 흐름
- 저장 버튼이 명확하고 닫기/취소와 구분되는가
- 저장 중 로딩 상태가 보이는가
- 저장 성공 후 목록 갱신, 상세 유지, 닫기 같은 결과가 예측 가능한가
- 중복 클릭으로 중복 저장될 가능성이 없는가

관련 법칙:
- Doherty 임계값
- Jakob의 법칙
- Peak-End 규칙

---

## 3. 모달

- 모달 제목이 작업 목적을 명확히 말하는가
- 닫기, 취소, 저장 버튼의 위치와 강도가 일관되는가
- 모달 안에서 스크롤이 필요하면 하단 액션이 사라지지 않는가
- 변경 사항이 있는 상태에서 닫을 때 손실 위험을 알려주는가

관련 법칙:
- 멘탈 모델
- Zeigarnik 효과
- Fitts의 법칙
- Peak-End 규칙

---

## 4. 일괄 처리/업로드

- 선택한 대상 수와 처리 범위가 명확히 보이는가
- 진행률, 성공 수, 실패 수가 표시되는가
- 실패 항목을 다시 확인하거나 재시도할 수 있는가
- 처리 중 화면을 닫거나 이동해도 되는지 알 수 있는가

관련 법칙:
- 목표 경사 효과
- Zeigarnik 효과
- Doherty 임계값
- 작업 기억

---

## 5. 빈 상태/오류 상태

- 데이터가 없을 때 원인과 다음 행동을 알려주는가
- 검색 결과 없음과 전체 데이터 없음이 구분되는가
- 권한 없음, 서버 오류, 네트워크 오류가 같은 문구로 뭉개지지 않는가
- 오류 상태에서 사용자가 복구할 수 있는 행동이 보이는가

관련 법칙:
- 능동적 사용자의 역설
- Peak-End 규칙
- Postel의 법칙

---

## 6. 피치 UI 산출물 금지/주의 패턴

아래 패턴은 UX 문제로 기록한다.

- 핵심 버튼과 위험 버튼이 같은 색/강도로 나열됨
- 검색 필터가 과도하게 많고 고급 조건 분리가 없음
- 테이블 컬럼이 많아 핵심 값이 뒤로 밀림
- 저장/검색/업로드 중 피드백이 없음
- 저장 성공 후 결과가 불명확함
- 모달 닫기 시 변경 손실 가능성을 알리지 않음
- 같은 상태값이 화면마다 다른 색이나 라벨로 표현됨
- 설명 텍스트가 많지만 실제 첫 행동은 불명확함

---

## 7. 최소 통합 체크 후보

추후 `proto-ui-qa`, `frontend-qa`, `team-e2e`에 통합한다면 아래 5개만 먼저 후보로 삼는다.

1. 핵심 액션이 보조 액션과 명확히 구분되는가
2. 위험 액션이 일반 액션과 색, 위치, 확인 흐름에서 구분되는가
3. 저장/검색/업로드 클릭 직후 로딩, 비활성화, 진행 상태가 보이는가
4. 오류 메시지가 필드 근처에 있고 다음 행동을 알려주는가
5. 작업 완료 후 결과와 다음 행동이 명확한가

이 5개는 기계 검증 실패가 아니면 `REJECTED`가 아니라 `CONDITIONAL` 또는 권고로 보고한다.

## 8. 오용 방지 규칙

| 상황 | 금지할 해석 | 올바른 해석 |
|------|-------------|-------------|
| 필터가 많음 | 무조건 접기 | 자주 쓰는 조건은 노출하고 드문 조건만 접기 |
| 버튼이 작음 | 모든 버튼 확대 | 조작 빈도와 위험도 기준으로 크기 조정 |
| 완료 피드백 부족 | 매번 모달 추가 | 반복 업무를 방해하지 않는 토스트/상태 갱신 우선 |
| 항목이 7개 초과 | Miller 법칙 위반 | 숫자보다 의미 단위 그룹화 여부 확인 |
| 화면이 단조로움 | 장식 추가 | 스캔성, 일관성, 주요 액션 구분을 먼저 확인 |
| 위험 액션 존재 | 확인 절차 무조건 추가 | 삭제/취소불가/대량 처리처럼 손실 비용이 큰 경우만 강화 |

---

## 9. 우선순위 판단

### 높음
- 잘못된 데이터 저장 가능성
- 위험 액션 오작동 가능성
- 저장/검색/업로드 결과 불명확
- 필수 업무 흐름 중단

### 중간
- 반복 업무 시간이 늘어남
- 화면 이해 비용이 큼
- 검색/필터/목록 스캔이 어렵다
- 모달/폼 조작 부담이 크다

### 낮음
- 일관성 부족
- 시각적 정돈 부족
- 보조 안내 문구 개선
- 장기적으로 통일하면 좋은 패턴

## 10. 리뷰 질문 풀

- 이 화면의 핵심 작업은 5초 안에 파악되는가?
- 사용자가 지금 해야 할 1순위 액션이 보조 액션과 명확히 구분되는가?
- 검색 조건, 선택 대상, 편집 중인 데이터의 맥락을 사용자가 기억해야 하는가?
- 저장/검색/업로드 클릭 직후 로딩, 비활성화, 진행 상태가 보이는가?
- 저장 성공/실패 후 데이터가 어떻게 바뀌었는지 확인 가능한가?
- 위험 액션은 일반 액션과 색, 위치, 확인 흐름에서 구분되는가?
- 테이블에서 가장 자주 확인하는 컬럼이 앞쪽에 있고 빠르게 스캔되는가?
- 필터가 많다면 기본 조건과 고급 조건이 분리되어 있는가?
- 오류 메시지는 문제 원인뿐 아니라 다음 행동을 알려주는가?
- 모달을 닫거나 이동할 때 미저장 변경 손실 가능성이 표시되는가?
- 같은 상태, 같은 버튼, 같은 흐름이 화면마다 같은 명칭과 시각 언어를 쓰는가?
- 법칙 적용이 실제 작업 성공률을 높이는가, 아니면 단순 취향/장식 지적인가?
