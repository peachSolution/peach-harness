# UX 리뷰 보고서 템플릿

아래 형식으로 보고한다. 문제 수는 3~7개를 기본으로 하며, 심각도 높은 항목을 먼저 둔다.

```markdown
## UX 검토 결과

한줄 결론: {대상 화면의 UX 상태를 한 문장으로 요약}

### 검토 범위
- 대상: {URL/파일/폴더/화면명}
- 주요 사용자: {확인된 사용자 또는 추정}
- 핵심 작업: {검색/등록/수정/검토/일괄 처리 등}
- 근거 자료: {확인한 파일, 스크린샷, Spec, DOM 등}

### 주요 문제

| 우선순위 | 심각도 | 문제 | 근거 UX 법칙 | 수정 방향 |
|----------|--------|------|-------------|-----------|
| 1 | 높음 | ... | ... | ... |
| 2 | 중간 | ... | ... | ... |

### 판정
- UX 판정: REJECTED 후보 / CONDITIONAL 후보 / 권고 / 특이 문제 없음
- 판정 근거: {높음/중간/낮음 문제 수와 근거 레벨}
- 주의: 이 판정은 기존 QA 스킬의 기계 검증 실패를 의미하지 않음

### 상세 의견

#### 1. {문제 제목}
- 심각도: 높음/중간/낮음
- 근거 레벨: 코드 확인 / DOM 확인 / 스크린샷 확인 / Spec 확인 / 추정
- 근거: {화면 또는 코드에서 확인한 사실}
- UX 법칙: {한글명} ({English Name})
- 영향: {사용자에게 생기는 문제}
- 제안: {구체적 수정안}

### 후속 전달

| 항목 | 내용 |
|------|------|
| 대상 스킬 | `peach-gen-ui-proto` / `peach-team-dev` / `peach-team-e2e` / 수동 보류 |
| 권장 owner | proto-ui-dev / ui-dev / frontend-qa / e2e-suite-qa / 사용자 판단 |
| 수정 범위 | 파일/화면/흐름 단위로 기재 |
| E2E 검증 포인트 | 클릭 후 피드백, 완료 결과, 실패 복구 등 |
| 보류 사유 | 업무 빈도/사용자 역할/실데이터 확인 필요 시 기재 |

### 적용 순서
1. {즉시 반영할 항목}
2. {다음 작업에서 반영할 항목}
3. {추후 디자인 시스템/공통 컴포넌트로 정리할 항목}

### 특이 문제 없음
- {문제 없는 검토 항목이 있으면 짧게 기록}

### 남은 확인 사항
- {실행 화면, 사용자 역할, 실제 데이터 등 추가 확인이 필요한 항목}
```

---

## 짧은 보고 형식

사용자가 빠른 검토를 요청하면 아래 형식을 사용한다.

```markdown
UX 관점에서 가장 먼저 볼 문제는 {핵심 문제}입니다.

1. {문제} — {법칙}, {수정 제안}
2. {문제} — {법칙}, {수정 제안}
3. {문제} — {법칙}, {수정 제안}

우선순위는 {1순위 수정}입니다.
```
