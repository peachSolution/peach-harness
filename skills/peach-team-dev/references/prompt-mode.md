# prompt 모드 동작

`peach-team-dev`가 mode/proto 없이 자연어 입력을 받았을 때 수행하는 절차.

## 목적

- Spec/proto 없이도 즉흥적 작업 가능하게 함 (버그 수정, 작은 기능 추가, 핫픽스)
- 변경 규모를 자동 추정해 적절한 모드(단독/팀)로 분기
- 큰 변경은 Spec 작성을 권고해 검증 기준 외부화 원칙 유지

## 입력 형태

```bash
/peach-team-dev "버그/개선 사항 자연어 설명" [force=Y] [model=...]
```

예:
```bash
/peach-team-dev "list.vue 검색 후 페이지네이션 클릭 시 검색어 초기화되는 문제 수정"
/peach-team-dev "게시판 상세 화면에 인쇄 버튼 추가하고 인쇄 페이지도 만들어줘"
/peach-team-dev "주문 모듈에 환불 기능 전체 추가" force=Y
```

## 절차 (4단계)

### 1. 입력 분석

#### 작업 규모 추정

다음 신호를 종합해 소/중/대 판정.

| 신호 | 소규모 | 중규모 | 대규모 |
|------|--------|--------|--------|
| 변경 파일 수 | 1개 | 2~5개 | 6개 이상 |
| 작업 유형 | 버그/스타일/검증 | 기능 추가/수정 | 새 모듈/스키마 변경 |
| 영향 모듈 수 | 1개 | 1개 | 2개 이상 |
| 신규 vs 수정 | 수정 | 수정 또는 부분 신규 | 신규 |
| DB 스키마 변경 | 없음 | 없음 | 있음 |
| 비즈니스 로직 복잡도 | 단순 | 중간 | 복잡 |

규모 판정은 prompt 텍스트 + 파일 경로 + 도메인 키워드를 보고 AI가 종합 판단한다. 임계치는 v1에서는 명시하지 않고 실사용 후 `peach-skill-feedback`으로 누적된 데이터로 v1.18.x 패치에서 정교화한다.

#### 영향 범위 추정

| 신호 | 판정 |
|------|------|
| `*.vue`, `pages/`, `modals/`, UI 컴포넌트 키워드 | UI |
| `controller`, `service`, `dao`, API 키워드 | backend |
| `store.ts`, Pinia 키워드 | store |
| 위 두 가지 이상 또는 "전체", "풀스택" 키워드 | fullstack |

#### 모듈 후보 식별

prompt에서 다음을 추출:
- 명시된 파일 경로 (예: `list.vue` → `front/src/modules/{X}/pages/list.vue` 검색)
- 도메인 키워드 (예: "게시판" → `board`, `notice` 등 매칭)
- 기존 modules 폴더 목록과 대조해 후보 1~3개 산출

#### 기존 기능 수정 감지

prompt에 "수정", "버그", "고쳐줘", "안 됨", "에러" 등이 포함되면 기존 기능 수정으로 판단.
이 경우 `docs/기능별설명/` 폴더에 해당 기능 As-Is 문서가 있는지 확인 후 권고.

### 2. 모듈명 확정 (사용자 확인)

분석 결과로 다음 형식의 안내를 출력한다.

```
입력: "list.vue 검색 후 페이지네이션 클릭 시 검색어 초기화되는 문제 수정"

분석 결과:
- 규모: 소규모 (단일 파일 추정)
- 영향: UI
- 작업 유형: 버그 수정

모듈 후보:
1. board (front/src/modules/board/pages/list.vue 존재)
2. notice (front/src/modules/notice/pages/list.vue 존재)
3. member (front/src/modules/member/pages/list.vue 존재)

어느 모듈인가요? (번호 또는 직접 입력)
```

선택 처리:
- 번호 입력 → 해당 후보 채택
- 직접 입력 → 입력값 검증 후 채택
- 단일 후보가 명확하면 자동 채택 안내 후 진행 ("board 모듈로 자동 채택합니다. 다르면 입력해주세요.")
- 후보 0개 → "해당 파일/모듈을 찾을 수 없습니다. 모듈명을 직접 입력해주세요."

### 3. 분기

규모 판정에 따라 다음 중 하나로 진입한다.

#### 소규모 → 단독 모드

- 팀 생성 생략
- 영향 범위에 맞는 단일 dev-agent만 호출
  - UI → ui-dev
  - backend → backend-dev
  - store → store-dev
- 해당 qa-agent로 검증 (랄프루프 5회)

#### 중규모 → 팀 모드 + Spec 자동 생성 제안 (선택)

- 팀 구성은 표준 모드와 동일 (영향 범위에 맞는 dev + qa)
- 사용자에게 Spec 자동 생성 제안:

```
중규모 변경입니다. Spec 문서를 자동 생성할까요?
(a) 자동 생성 — prompt 텍스트와 분석 결과로 docs/spec/.../{모듈}-{날짜}.md 자동 생성
(b) 생략 — Spec 없이 prompt 텍스트만으로 진행
(c) /peach-gen-spec 직접 호출 — 대화형 Spec 작성 후 재진입
```

- 자동 생성 선택 시 prompt + 분석 결과 + 기존 모듈 컨텍스트로 간단한 Spec md 생성
- 이후 표준 모드 흐름 진행

#### 대규모 → Spec 권고

```
⚠️ 큰 변경으로 추정됩니다 (분석 결과: [근거]).

검증 기준 외부화 원칙상 Spec 작성을 권고합니다:
(a) /peach-gen-spec 호출 → Spec 작성 후 재진입 (권장)
(b) force=Y로 강제 진행 — Spec 없이 진행하되 검증 한계를 인지

어떻게 할까요?
```

- (a) 선택 시: 사용자가 별도로 `/peach-gen-spec` 호출. 본 호출은 종료.
- (b) 선택 또는 `force=Y` 인자 있음: 중규모 흐름과 동일하게 진행하되 완료 보고에 "검증 한계: 대규모 변경을 Spec 없이 진행함" 명시.

### 4. 실행

규모별 분기에 따라 서브에이전트에 다음 컨텍스트를 주입한다.

```
## 작업 (prompt 모드)

원본 prompt:
[원본 텍스트]

분석 결과:
- 규모: [소/중/대]
- 영향: [backend/ui/store/fullstack]
- 모듈: [확정된 모듈명]
- 작업 유형: [버그/기능추가/리팩토링 등]

검증 기준:
- prompt 텍스트 자체
- (중/대규모 + 자동 생성 Spec 있을 때) 자동 생성 Spec 경로
- 기존 코드 회귀 방지 (테스트 통과)

검증 한계:
- prompt 모드는 검증 기준이 약합니다.
- 결과를 사용자가 직접 확인해야 합니다.

지시:
test-data 가이드 코드 패턴을 따라 작업하되, prompt에 명시된 변경만 수행합니다.
요청되지 않은 추가 리팩토링이나 인접 코드 수정은 하지 않습니다.
```

## 랄프루프 상한

prompt 모드는 검증 기준이 약하므로 **5회를 상한**으로 둔다 (표준 모드 10회보다 짧음).

5회 초과 시:
```
⚠️ Ralph Loop 5회 초과

prompt 모드의 검증 기준 한계로 더 이상 자동 수정이 어렵습니다.
다음 중 선택해주세요:
(a) /peach-gen-spec으로 Spec 작성 후 재진입
(b) AI Plan 모드 + 직접 수정으로 전환
(c) 현재 상태로 중단하고 사용자가 직접 검토
```

## 에러 케이스

| 상황 | 처리 |
|------|------|
| prompt가 너무 짧거나 모호 (예: "고쳐줘") | 사용자에게 구체화 요청 |
| 모듈 후보 0개 | 모듈명 직접 입력 요청 |
| 규모 판정 모호 (소-중 경계, 중-대 경계) | 사용자에게 확인 (예: "이 변경이 단일 파일 수정인가요, 여러 파일에 걸친 변경인가요?") |
| 영향 범위 모호 | 사용자에게 확인 (backend/UI/풀스택 중 선택) |

## 비고

- prompt 모드는 **검증 기준 외부화 원칙의 예외**다. 본질적으로 즉흥적 작업이므로 검증 강도가 약하다.
- 큰 변경은 자동으로 Spec 권고가 뜨므로, 박동재님은 작은 변경에서만 prompt 모드를 자유롭게 쓰면 된다.
- 매우 작은 수정(예: 오타 1개)은 prompt 모드보다 AI Plan 모드 + Edit이 더 빠르다. 사용자가 적절히 판단.
