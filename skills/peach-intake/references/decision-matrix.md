# Intake 판단표

`peach-intake`가 작업 규모와 후속 스킬 경로를 판단할 때 사용하는 기준이다.

## 작업 유형

| 유형 | 신호 | 기본 경로 |
|------|------|-----------|
| 작은 버그 | 특정 파일/증상/재현 조건이 명확하고 DB 변경 없음 | 직접 수정 또는 `peach-team-3a` |
| 기존 기능 수정 | 기존 모듈 구조 파악이 필요함 | `peach-doc-feature` 후 직접 수정 또는 `peach-team-dev` |
| 신규 기능 | 새 화면/API/DB/상태값이 필요함 | `peach-gen-spec`부터 시작 |
| PRD-first | PRD만 있고 구현 기준이 없음 | `peach-gen-spec`, 필요 시 DB 후보 검토 |
| DB 중심 | 테이블/컬럼/인덱스/상태값 변경이 핵심 | `peach-gen-db` 또는 `peach-db-migrate` |
| UI 중심 | 화면 해석과 기획 검토가 핵심 | `peach-team-ui-proto` |
| 검증 중심 | 이미 구현된 기능의 흐름 확인 | `peach-team-e2e` |

## Spec 필요 판정

Spec이 필요하거나 보강되어야 하는 경우:

- 요구사항이 PRD나 회의 메모 수준이다.
- 권한 경계가 있다.
- 상태 전이가 있다.
- 오류 케이스가 중요하다.
- API 요청/응답 기대값이 필요하다.
- TEST_ID로 구현/TDD/E2E 추적이 필요하다.
- PRD에는 있으나 Spec에 없는 요구가 발견됐다.

Spec 없이 진행 가능한 경우:

- 단일 파일 수정이다.
- 기존 Spec이나 기능 문서가 충분하다.
- 비즈니스 규칙 변경이 없다.
- 변경 결과를 test/lint/build로 검증할 수 있다.

## DB 필요 판정

DB 단계가 필요한 경우:

- 신규 테이블이 필요하다.
- 컬럼/인덱스/상태값이 추가된다.
- 이력, 로그, 관계 구조가 새로 필요하다.
- Spec 구현 중 schema 부재가 예상된다.

DB 판단을 보류해야 하는 경우:

- 상태값 후보가 확정되지 않았다.
- 데이터 소유 범위가 모호하다.
- PRD와 Spec/schema가 충돌한다.
- 기존 테이블 재사용 여부를 사용자 결정이 필요하다.

## ui-proto 필요 판정

ui-proto가 필수 또는 강하게 권장되는 경우:

- 신규 화면이 많다.
- 다단계 폼, 다중 모달, 탭, 차트, 복잡 검색이 있다.
- 기획자가 화면으로 먼저 검토해야 한다.
- E2E가 화면 흐름을 기준으로 시나리오를 나눠야 한다.
- Spec 텍스트만으로 인터랙션 해석이 어렵다.

ui-proto를 생략할 수 있는 경우:

- 백엔드 중심 기능이다.
- 기존 화면에 API/Store만 연결한다.
- 단순 CRUD이고 Spec의 화면 흐름 요약이 충분하다.
- 작은 버그 수정이다.

## E2E 필요 판정

E2E가 필요한 경우:

- 사용자가 보는 저장/수정/삭제 결과를 확인해야 한다.
- 권한별 UI 차이가 있다.
- 다중 화면 흐름이 있다.
- ui-proto 부합 검증이 필요하다.
- 릴리스 전 최종 검증이다.

E2E를 선택으로 둘 수 있는 경우:

- 단일 API 수정이다.
- TDD와 타입/빌드 검증으로 충분하다.
- 화면 흐름 변화가 없다.
