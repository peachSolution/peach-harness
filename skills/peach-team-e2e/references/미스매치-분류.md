## 목적

- 실패 원인을 명확히 분류해 잘못된 보완 시도 방지 (코드 문제인데 시나리오를 고치는 등)
- 자동 보완 가능한 항목과 사용자 확인 필요 항목 구분
- 랄프루프의 효율 극대화

## 5가지 분류

| 분류 | 정의 | 보완 경로 |
|------|------|----------|
| **(a) Spec 비즈니스 규칙 위반** | 본 프로젝트 코드가 Spec과 다르게 동작 | 코드 수정 (peach-team-dev) |
| **(b) ui-proto 화면 흐름과 다름** | 본 프로젝트 화면이 ui-proto와 다르게 구현 | 코드 수정 (peach-team-dev mode=ui) |
| **(c) 시나리오 자체 오류** | 시나리오 코드의 셀렉터/로직 오류 | 시나리오 수정 (peach-e2e-scenario 자동수정) |
| **(d) PRD 해석 누락** | PRD에는 있으나 Spec/ui-proto 검증 기준에 없는 요구 | `PRD_TO_SPEC_REQUIRED`로 Spec 보강 |
| **(e) 기준 모호** | 검증 기준 부족 또는 Spec/ui-proto 충돌 | 사용자 확인 또는 기준 보강 |

## 분류 판단 트리

```
실행 실패 또는 검증 미부합
       │
       ├─ 검증 기준(ui-proto/Spec)에 명시된 동작과 다른가?
       │      │
       │      ├─ Yes → (a) 또는 (b)
       │      │       │
       │      │       ├─ 화면 레이아웃/인터랙션 흐름 차이 → (b)
       │      │       └─ 비즈니스 규칙/데이터 차이 → (a)
       │      │
       │      └─ No (검증 기준에는 맞지만 실패) → (c) 또는 모호
       │              │
       │              ├─ 셀렉터/타이밍/문법 문제 → (c)
       │              └─ 모호 → (d)
       │
       ├─ PRD 원천에는 있으나 Spec/ui-proto에 없는 요구인가? → (d)
       │
       └─ 검증 기준 자체가 모호 → (e)
```

## 분류별 신호

### (a) Spec 비즈니스 규칙 위반의 신호

- API 응답값이 Spec과 다름 (필드 누락, 타입 불일치)
- 검증 메시지가 Spec과 다름 ("이메일 형식이 올바르지 않습니다" vs "잘못된 이메일")
- 권한 분기 누락 (관리자만 보여야 하는데 일반 사용자도 보임)
- 상태 전이 누락 (DRAFT → PUBLISHED 인데 DRAFT에서 PUBLISH 버튼 안 보임)
- 에러 케이스 처리 누락 (네트워크 오류 시 안내 메시지 없음)
- DB 상태가 Spec과 다름 (UPDATE 후 expected 값과 actual 값 불일치)

→ 본 프로젝트 코드 수정 필요. 자동 위임 안전 조건을 만족하면 `peach-team-dev`에 바로 위임한다.

### (b) ui-proto 화면 흐름과 다름의 신호

- 컴포넌트 배치가 ui-proto와 다름 (검색 폼 위치, 버튼 순서)
- 화면 전환 흐름이 ui-proto와 다름 (목록에서 상세로 가는 방식 차이)
- 모달 vs 페이지 차이 (ui-proto는 모달인데 본 프로젝트는 별도 페이지)
- 표시되는 필드 차이 (ui-proto에는 있는데 본 프로젝트에는 없음)
- 색상/스타일 차이는 (b)에 포함하지 않음 (디자인 시스템 차이는 별도)

→ 본 프로젝트 UI 코드 수정 필요. 자동 위임 안전 조건을 만족하면 `peach-team-dev mode=ui`에 바로 위임한다.

### (c) 시나리오 자체 오류의 신호

- 셀렉터 timeout (요소 못 찾음, 검증 기준에는 맞게 화면이 그려져 있음)
- locator not found (실제 DOM은 있는데 셀렉터가 잘못됨)
- 타이밍 문제 (waitForTimeout 후 즉시 실패, 비동기 대기 부족)
- URL 패턴 timeout (실제 URL은 맞는데 정규식 오류)
- popup 닫힘 감지 실패 (waitForEvent 사용법 오류)
- 시나리오 내 변수/환경변수 전달 오류

→ 시나리오 코드 수정. **자동수정 가능** — peach-e2e-scenario 자동수정 패턴 적용 (3회).

### (d) PRD 해석 누락의 신호

- PRD에는 명시되어 있으나 Spec `TEST_ID`에 매핑되지 않은 요구
- PRD의 화면/정책/상태값 요구가 ui-proto와 Spec 모두에 없음
- 사용자 검토 중 "PRD에는 있는데 구현에 없다"는 항목이 발견됨

→ 코드 수정이 아니라 `PRD_TO_SPEC_REQUIRED`로 보고한다. Spec 보강 후 다시 dev/e2e를 실행한다.

### (e) 기준 모호의 신호

- 실패는 명확하지만 검증 기준에 해당 항목이 명시되지 않음
- 검증 기준이 추상적 ("적절히 처리한다", "사용자 친화적으로")
- ui-proto와 Spec이 충돌하는 항목
- 권한 정책, DB 구조, 공통 컴포넌트 변경이 필요한 항목

→ 사용자 확인 또는 기준 보강이 필요하다. 자동 위임하지 않는다.

## 모호한 케이스 처리

### 분류 불가 신호

- 실패는 명확하지만 검증 기준에 해당 항목이 명시되지 않음
- 검증 기준이 추상적 ("적절히 처리한다", "사용자 친화적으로")
- ui-proto와 Spec이 충돌하는 항목

### 처리

```markdown
## 모호한 미스매치 보고

증상: [실제 실패 내용]
검증 기준 확인 결과:
- ui-proto: [해당 항목 명시 여부]
- Spec: [해당 항목 명시 여부]

가능한 해석:
1. (a) Spec 위반으로 추정: [근거]
2. (b) proto 불일치로 추정: [근거]
3. (c) 시나리오 오류로 추정: [근거]
4. (d) PRD 해석 누락으로 판단: [근거]
5. (e) 기준 모호로 판단: [근거]

어떻게 처리할까요?
- (a)/(b) 코드 수정으로 진행
- (c) 시나리오 수정으로 진행
- (d) PRD_TO_SPEC_REQUIRED로 Spec 보강
- (e) 검증 기준 보강 후 재시도 (Spec/proto 갱신)
```

## 보완 경로별 동작

### (a)/(b) 본 프로젝트 코드 수정

```
1. 미스매치 근거(Spec TEST_ID 또는 ui-proto 화면)를 확인
2. 자동 위임 안전 조건 확인
3. 안전 조건 만족 시:
   - 작은 수정: /peach-team-dev "수정 prompt"
   - UI 수정: /peach-team-dev [모듈] mode=ui proto=<경로>
   - backend/store 수정: /peach-team-dev [모듈] mode=backend
4. 안전 조건 불만족 시 (e) 기준 모호로 보고
5. 수정 완료 후 e2e 재실행
6. 여전히 미스매치이면 랄프루프 카운트 +1
```

자동 위임 안전 조건:
- Spec 또는 ui-proto 근거가 명확함
- 수정 대상 레이어가 분명함
- DB 구조, 공통 모듈, 권한 정책 변경 없음
- 수정 후 TDD/lint/build/E2E 재실행 가능

### (c) 시나리오 수정

```
1. peach-e2e-scenario 자동수정 패턴 적용
   - 셀렉터 불일치 → agent-browser eval로 실제 DOM 확인 → 수정
   - 타이밍 문제 → waitForEvent로 변환
   - URL 패턴 → 실제 URL 확인 → 정규식 수정
2. 수정 후 재실행 (최대 3회)
3. 3회 실패 시 (c) 분류 신뢰도 재검토 → 사용자에게 보고
```

자동수정 상세는 `peach-e2e-scenario/references/자동수정-판단트리.md` 참조.

## 분류 결과 누적

랄프루프 동안 분류 이력을 누적해 보고서에 포함한다.

```markdown
## 미스매치 분류 이력

| 회차 | 분류 | 증상 | 처리 |
|------|------|------|------|
| 1 | (c) | 셀렉터 timeout | 자동수정 (셀렉터 변경) |
| 2 | (a) | API 응답 필드 누락 | 코드 수정 위임 |
| 3 | (b) | 모달 vs 페이지 차이 | 코드 수정 위임 |
| 4 | (c) | 타이밍 문제 | 자동수정 (waitForEvent) |
| 5 | (d) | PRD 요구가 Spec에 없음 | PRD_TO_SPEC_REQUIRED |
| 6 | (e) | Spec/proto 충돌 | 사용자 확인 |
```

이 이력은 완료 보고에 포함되며, `docs/qa/{년}/{월}/[기능명]-e2e-{날짜}.md`에 저장된다.

## 비고

- 분류는 e2e-suite-qa 에이전트가 1차 판단하고, 모호하면 오케스트레이터(이 스킬)가 (d)로 분리해 사용자에게 보고
- 분류 정확도는 검증 기준의 명확성에 비례. Spec/ui-proto가 정확할수록 분류가 명확해짐
- 분류 신뢰도가 낮으면 자동 보완을 시도하지 말고 사용자에게 보고
