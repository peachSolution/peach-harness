# E2E QA 판정과 완료 정책

`peach-team-e2e`의 QA 판정, 보완 루프, 완료 보고 기준이다.

## 판정

| 판정 | 의미 | 처리 |
|------|------|------|
| `APPROVED` | 단위 시나리오 + 통합 suite 통과, 검증 기준 부합 | 증거 수집 후 완료 |
| `CONDITIONAL` | 일부 한계가 있으나 릴리스 판단이 가능한 상태 | 오케스트레이터가 근거를 기록하고 수용/보류 판단 |
| `REJECTED` | 검증 기준 위반 또는 자동수정 불가 | 미스매치 분류 후 보완 루프 |

## 보완 루프

| 분류 | 처리 |
|------|------|
| Spec 비즈니스 규칙 위반 | 안전 조건 만족 시 `peach-team-dev` 자동 위임 |
| ui-proto 화면 흐름 불일치 | 안전 조건 만족 시 `peach-team-dev mode=ui` 자동 위임 |
| 시나리오 오류 | `peach-e2e-scenario` 자동수정 패턴으로 최대 3회 수정 |
| PRD 해석 누락 | `PRD_TO_SPEC_REQUIRED`로 Spec 보강 후 재시도 |
| 기준 모호 | 사용자 확인 또는 Spec/ui-proto 보강 후 재시도 |

상한은 단순 시나리오 수정 3회, 소규모 부합 검증 5회, 중대규모 부합 검증 10회다. 상한 도달 시 미스매치 이력과 권고를 보고하고 중단한다.

## 자동 위임 안전 조건

다음 조건을 모두 만족하면 사용자 확인 없이 `peach-team-dev`에 수정 프롬프트를 위임한다.

- Spec 또는 ui-proto에 명확한 근거가 있다.
- 수정 대상 레이어가 backend/store/ui 중 하나로 분명하다.
- DB 구조, 공통 모듈, 권한 정책을 변경하지 않는다.
- 기존 Must Follow를 침범하지 않는다.
- 수정 후 TDD/lint/build/E2E 재실행이 가능하다.

하나라도 만족하지 않으면 `기준 모호` 또는 `사용자 확인 필요`로 보고한다.

## 완료 보고

완료 보고에는 다음을 포함한다.

- 검증 기준: proto 경로, Spec 사본 경로, Spec-only 여부, 검증 신뢰도
- 생성 산출물: 단위 시나리오 목록, 통합 suite md, 실행 보고 경로
- 실행 결과: 단위 시나리오와 suite 통과 여부
- 부합 결과: 화면 흐름, 인터랙션, 비즈니스 규칙, 데이터 정확성, 에러/예외
- Spec `TEST_ID`와 검증 상태(`V01/V02/V03/V80/V90`) 매핑, 상태 요약 재계산 결과
- 미스매치 이력: 분류, 원인, 수정 방식, 재검증 결과
- 자동 위임 이력: 위임 대상, 수정 프롬프트 요약, team-dev 검증 결과
- PRD_TO_SPEC_REQUIRED 발생 여부와 Spec 보강 필요 항목
- 검증 불가 또는 약한 기준: Spec/proto 부재, 자연어 기준 사용, UI Proto 부재로 약한 시각 검증, 사용자 확인 필요 항목

Spec-only 모드에서는 신뢰도 `중간`과 시각 검증 한계를 명시한다. 검증 기준 부재 모드에서는 신뢰도 `낮음`과 사후 Spec 작성을 권고한다.
