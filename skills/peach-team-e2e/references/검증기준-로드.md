`peach-team-e2e`가 ui-proto와 Spec에서 검증 기준을 추출하는 방법.

## 목적

- 코드 동작 검증을 넘어 **기획 의도 부합 검증**의 기반을 만든다
- ui-proto의 시각/흐름 + Spec의 비즈니스 규칙을 단일 검증 컨텍스트로 통합
- 검증 우선순위 규칙을 명확히 적용

## 입력

| 모드 | 입력 |
|------|------|
| 표준 검증 | `proto=<경로>` 인자 + 본 프로젝트 `docs/spec/...` |
| Spec-only | 본 프로젝트 `docs/spec/...` 만 |
| 검증 기준 부재 | 사용자 자연어 입력 |

## 표준 검증 모드 절차

### 1. ui-proto 경로 검증

```bash
# 폴더 존재
[ -d "$PROTO_PATH" ] || error "proto 경로 없음"

# _spec.md 존재
[ -f "$PROTO_PATH/_spec.md" ] || error "_spec.md 없음"

# 화면 폴더 확인
find "$PROTO_PATH" -name "*.vue" -type f | head -20
```

### 2. 본 프로젝트 Spec 사본 확인

```bash
# peach-team-dev에서 자동 복사된 Spec 위치 확인
ls docs/spec/{년}/{월}/

# 두 Spec 비교 (불일치 시 사용자 확인)
diff "$PROTO_PATH/_spec.md" docs/spec/{년}/{월}/{기능명}.md 2>/dev/null
```

차이가 발견되면:
```
⚠️ ui-proto의 _spec.md와 본 프로젝트의 Spec이 다릅니다.

차이 요약:
[diff 결과 일부]

어떻게 처리할까요?
(a) ui-proto _spec.md를 1차 기준으로 사용 (권장)
(b) 본 프로젝트 Spec을 1차 기준으로 사용
(c) 두 Spec을 합쳐서 사용 (수동 병합 필요)
```

### 3. 검증 기준 컨텍스트 추출

#### 시각/흐름 기준 (ui-proto에서 추출)

```bash
# overview 화면 (기획서)
cat "$PROTO_PATH/overview/pages/overview.vue"

# 서브모듈 화면 목록
ls "$PROTO_PATH/"   # 폴더 목록
find "$PROTO_PATH" -name "*.vue" -type f | grep -v overview
```

추출 항목:
- **페이지 진입점**: 사용자가 처음 보는 화면
- **사용자 액션 시퀀스**: 클릭 → 입력 → 저장 등의 순서
- **화면 전환 흐름**: 페이지 → 모달 → 페이지 같은 전환 패턴
- **컴포넌트 배치**: 검색 폼 → 테이블 → 페이지네이션 같은 레이아웃
- **결과 확인 화면**: 작업 후 사용자가 보는 결과 화면

#### 비즈니스 규칙 기준 (Spec에서 추출)

`_spec.md`의 다음 섹션을 파싱:
- 입력 검증 규칙 (필수 필드, 형식, 길이 등)
- 권한 분기 (관리자만 보기, 일반 사용자 차단 등)
- 상태 전이 (DRAFT → PUBLISHED 같은 상태 머신)
- 에러 케이스 (서버 오류, 네트워크 오류, 검증 실패 시 메시지)
- API 엔드포인트와 응답 형태
- DB 상태 변화 기대값

#### 데이터 정확성 기준 (Spec에서 추출)

- API 응답 구조와 필드 타입
- DB 테이블의 변경 기대값 (UI/API/TDD fixture 수행 후 SELECT로 확인할 최종 상태)
- 외부 시스템 연동 시 응답 형태

### 4. 컨텍스트 주입

다음 형식으로 e2e-scenario-dev, e2e-suite-dev 에이전트에게 전달:

```markdown
## 검증 기준 (표준 모드)

### 1. ui-proto 화면 (시각/흐름 1차 기준)
경로: $PROTO_PATH/

페이지 진입점: [추출된 진입점]

사용자 액션 시퀀스:
1. [액션 1]
2. [액션 2]
...

화면 전환 흐름:
- [전환 1]
- [전환 2]

### 2. Spec (비즈니스 규칙 1차 기준)
경로: docs/spec/{년}/{월}/{기능명}.md

입력 검증 규칙:
- [규칙 1]

권한 분기:
- [분기 1]

상태 전이:
- [상태 머신]

에러 케이스:
- [케이스 1]

### 3. 검증 우선순위 규칙
- 화면 레이아웃, 인터랙션 흐름: ui-proto 우선
- 비즈니스 규칙, 데이터 정확성: Spec 우선
- 둘 다 모호: 검증 불가로 보고
```

## Spec-only 검증 모드 절차

ui-proto가 없으면 Spec에서 화면 흐름까지 추출한다 (정확도 낮음).

### 1. Spec 위치 확인

```bash
ls docs/spec/{년}/{월}/
```

### 2. 화면 흐름 추정

Spec에서 다음을 추출 (ui-proto만큼 정확하지는 않음):
- "사용자가 X 화면에서 Y 액션을 한다" 같은 서술 → 액션 시퀀스
- "Z 화면이 나타난다" → 화면 전환
- 화면 레이아웃은 일반적인 CRUD 패턴 가정 (목록 → 상세 → 수정 → 저장)

### 3. 한계 명시

```markdown
## 검증 기준 (Spec-only 모드, 한계 있음)

### 1. 비즈니스 규칙 (Spec, 1차 기준)
[추출 결과]

### 2. 화면 흐름 (Spec에서 추정, 정확도 낮음)
[추정 결과]

### 한계
- ui-proto가 없어 시각 검증 불가
- 화면 레이아웃 검증은 일반적 패턴으로 가정
```

## 검증 기준 부재 모드 절차

Spec/proto 모두 없으면 사용자에게 자연어로 검증 기준을 입력받는다.

### 입력 형식

```
검증 기준이 없습니다. 어떻게 할까요?

(a) /peach-gen-spec으로 사후 Spec 작성 후 재진입 (권장)
(b) 자연어로 검증 기준 직접 입력
(c) 코드 동작 검증만 진행 (검증 기준 외부화 포기)
```

(b) 선택 시:
```
검증할 흐름을 자연어로 입력해주세요. 예시:
"로그인 → 게시판 작성 → 저장 → 목록에서 확인"
"회원가입 → 이메일 인증 → 로그인 → 마이페이지 진입"
```

### 자연어 입력 처리

자연어를 액션 시퀀스로 파싱:
- "→" 또는 줄바꿈으로 단계 분리
- 각 단계를 하나의 단위 시나리오로 변환
- 검증 포인트는 단계 끝에서 화면 상태 확인 (URL, 주요 텍스트 존재 여부)

### 한계 명시

```markdown
## 검증 기준 (자연어 입력 모드, 한계 큼)

### 입력
[원본 자연어]

### 파싱된 단계
[단계 목록]

### 한계
- 비즈니스 규칙 검증 불가
- 데이터 정확성 검증 불가
- 결과를 사용자가 직접 확인해야 함
```

## 충돌 해결

### Spec과 ui-proto가 다를 때

| 충돌 유형 | 해결 |
|----------|------|
| 화면 흐름 차이 | ui-proto 우선 (검증 우선순위 규칙) |
| 비즈니스 규칙 차이 | Spec 우선 |
| ui-proto에만 있는 화면 | 시각 검증만 수행, 비즈니스 검증 생략 |
| Spec에만 있는 규칙 | 시각 화면 없이 비즈니스 규칙만 검증 |

### 검증 불가 항목 처리

검증 기준이 모호하거나 불완전한 항목은 다음 형식으로 보고:

```
⚠️ 검증 불가 항목

[항목 1]:
- ui-proto: 명시 없음
- Spec: 모호함 ("적절히 처리한다" 같은 서술)
- 결정: 검증 생략, 사용자가 직접 확인 권고

[항목 2]:
- ...
```

## 비고

- 이 절차의 산출물(검증 기준 컨텍스트)은 **e2e-scenario-dev, e2e-suite-dev 에이전트에게 그대로 전달**된다
- 컨텍스트가 너무 길면 핵심만 요약 (각 항목당 3~5줄 이내)
- ui-proto의 화면 폴더는 직접 읽지 않고 **경로만 전달**해서 에이전트가 필요 시 직접 읽도록 한다 (토큰 절약)
